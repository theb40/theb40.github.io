Data aggregation

Problem: The raw data came in the form of 12 CSV files with the same structure. There is a limitation to the ability to handle multiple tables of SQL query, which made it reasonable to combine all 12 files into a single table.

Process:

  • Creating the table:
create table bikedata_2023 (
    ride_id nvarchar(50),
    bike_type nvarchar(50),
    started_date date,
    started_time time,
    end_date date,
    end_time time,
    start_station nvarchar(100),
    start_station_id nvarchar(50),
    end_station nvarchar(100),
    end_station_id nvarchar(50),
    ride_type nvarchar(10)
)
  • Importing data from other table: 
insert into bikedata_2023 (
    ride_id,
    bike_type,
    started_date,
    started_time,
    end_date,
    end_time,
    start_station,
    start_station_id,
    end_station,
    end_station_id,
    ride_type
)
SELECT 
    ride_id,
    rideable_type,
    CONVERT(date, LEFT([started_at],10)),
    CONVERT(time, RIGHT([started_at],16)),
    CONVERT(date, LEFT([ended_at],10)),
    CONVERT(time, RIGHT([ended_at],16)),
    start_station_name,
    start_station_id,
    end_station_name,
    end_station_id,
    member_casual
FROM
    [cyclistic].[dbo].[202301_divvy_tripdata]

Dropping unnecessary data

Problem: While inspecting the raw data, some entries with unexpectedly long station IDs became noticeable. After further checking, it was apparent that these entries came from Cyclisticâ€™s repairing stations or warehouses.

Process:

  • Checking station IDs length to understand the anomaly: 
SELECT TOP 20
    start_station,
    start_station_id,
    LEN(start_station_id) as len_id,
    COUNT(start_station) as appearance

FROM
    dbo.bikedata_2023

SELECT 
    *

Ride time inconsistency

Problem: In the process of setting up another query to further understand the differences between member and casual ride length, several entries came up as negative. Further inspection concluded that a majority of these entries got their start time and end time reversed, while some other were negative because their start time was very close to midnight, resulting in a reversed length output.

Process:

  • Inspecting the ride length:
WITH temp_data as(    
    SELECT  
        started_time,
        started_date,
        end_time,
        end_date,
        start_station,
        end_station,
        CASE 
            WHEN DATEDIFF(MINUTE, started_time,end_time) > 0 
                THEN DATEDIFF(MINUTE, started_time,end_time)
            WHEN DATEDIFF(MINUTE, started_time,end_time) < 0 
                THEN DATEDIFF(MINUTE, started_time,end_time) + 1440 
            ELSE 0 
        END AS use_time,
        bike_type,
        ride_type
    FROM 
        dbo.bikedata_2023
)
SELECT 
    DISTINCT use_time, 
    COUNT(ride_type) OVER (PARTITION BY use_time) as frequency
FROM 
    temp_data
ORDER BY 
    use_time DESC
This query normalized those entries that were close to midnight while also showed that those entries with start time and end time reversed would appear negative nevertheless.
  • Dropping entries with reversed time:
DELETE FROM 
    dbo.bikedata_2023
WHERE 
    DATEDIFF(MINUTE, started_time,end_time) + 1440 < 0

Coordinates inconsistency

Problem: The coordinates of the station were recorded using bikeâ€™s GPS data which created multiple different data points for a single station. This can be negated using the averaged coordinates of each station.

Process:

Using a raw data table with the most amount of entries, the average coordinates of each station can be extracted.
  • Extracting averaged coordinates: 
SELECT 
    start_station_name station,
    AVG(start_lat) latitude,
    AVG(start_lng) longitude
FROM 
    dbo.[202207_divvy_tripdata_cor] 
GROUP BY 
    start_station_name
This table was then saved separately as Divvy_Bicycle_Station.
